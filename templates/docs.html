Overview

The OPEN API Dictionary API is a REST API built with FastAPI (Python). It serves structured dictionary data from a local SQLite database, with a built-in AI-powered spell-correction fallback for unrecognised words.

Base URL

http://localhost:8000

Protocol

HTTP / REST

Response Format

application/json
api

Endpoint

GET /api/{language}/{word}
main.py

Retrieve the definition, part of speech, synonyms, and antonyms for a given word in the specified language.

PATH
{language} — 3-letter ISO 639-2 language code (e.g. eng). Must be one of the supported languages below.
PATH
{word} — The word to look up. Case-insensitive, leading/trailing whitespace is stripped.
GET /

Renders the home page (home.html) via Jinja2.

GET /docs

Renders this documentation page (docs.html).

data_object

Response Schema

Field Type Description
word string The (potentially corrected) word that was looked up.
results array Array of result objects. Empty array when not found.
→ definition string The word's meaning.
→ pos string Part of speech (e.g. n., v., adj.).
→ synonyms string | null Comma-separated synonyms, or null.
→ antonyms string | null Comma-separated antonyms, or null.
error string | null null on success; error message string on failure.

error

Error Handling

404-like Word Not Found

When the word (and spell-corrected variants) are absent from the database, the API returns 200 with an empty results array and a populated error field.

{
  "word": "zzzyyyxxx",
  "results": [],
  "error": "Word not found in the eng dictionary."
}
400-like Unsupported Language

When the language code is not in ALLOWED_LANGUAGES.

{
  "word": "hello",
  "results": [],
  "error": "Language 'deu' is not supported."
}
spellcheck

Spell Correction

When a word is not found in the database, the API automatically attempts to correct the spelling before returning a "not found" error. This is a two-step process:

1

Fast local correction

Uses Python's difflib.get_close_matches against the full in-memory word set (≈175k entries for English) with a similarity cutoff of 0.7.

2

LLM fallback

If local correction finds no match, the query is sent to Google Gemini 1.5 Flash via LangChain (ai.py) and asked to return the single most likely intended dictionary word.

lightbulb

The word field in the response will reflect the corrected spelling when a correction was applied, so your client can inform users of the substitution.
